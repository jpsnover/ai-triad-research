# Document Processing Pipeline — Technical Specification

This document describes the exact mechanisms that `Import-AITriadDocument` and its downstream functions use to ingest a document, chunk it, extract claims, map those claims to existing taxonomy nodes, and detect unmapped concepts. All parameter values, thresholds, and algorithm details are drawn from the implementation.

## Pipeline Overview

```
Document (PDF/DOCX/HTML/URL)
    |
    v
[1. INGEST] Convert to Markdown, extract metadata, create source directory
    |
    v
[2. SIZE CHECK] Estimate token count (chars / 4)
    |
    +--- <= 20,000 tokens ---> [3a. SINGLE-CALL SUMMARY]
    |                              Full document + taxonomy in one AI call
    |
    +--- > 20,000 tokens ----> [3b. CHUNKED SUMMARY]
                                   Split into chunks, summarize each, merge
    |
    v
[4. CLAIM EXTRACTION] AI extracts key_points, factual_claims, unmapped_concepts
    |
    v
[5. TAXONOMY MAPPING] AI maps each claim to existing nodes by ID
    |
    v
[6. UNMAPPED DETECTION] Claims with no node match become unmapped_concepts
    |
    v
[7. CONFLICT DETECTION] Cross-POV disagreements flagged automatically
```

## 1. Document Ingestion

**Entry point:** `Import-AITriadDocument` (`scripts/AITriad/Public/Import-AITriadDocument.ps1`)

### Input Methods

| Parameter Set | Source | Example |
|--------------|--------|---------|
| `-Url` | Web page or direct file URL | `Import-AITriadDocument -Url 'https://example.com/paper.pdf'` |
| `-File` | Local file path | `Import-AITriadDocument -File './paper.pdf'` |
| `-Inbox` | Batch import from inbox directory | `Import-AITriadDocument -Inbox` |

### Format Conversion

Documents are converted to Markdown via `DocConverters.psm1`. Each format has a priority chain of conversion tools — the first available tool wins.

| Format | Priority Chain | Notes |
|--------|---------------|-------|
| PDF | `markitdown` > `pdftotext` > `mutool` > placeholder | markitdown preferred for layout preservation |
| DOCX | `markitdown` > `pandoc` > ZIP/XML fallback | ZIP fallback extracts raw XML text |
| HTML (local) | `markitdown` > built-in converter | Built-in strips non-content elements |
| HTML (URL) | Direct fetch + built-in converter | Detects PDF links and downloads directly |
| PPTX/XLSX/CSV/EPUB | `markitdown` only | Returns null if markitdown unavailable |

**Built-in HTML converter** strips: `<script>`, `<style>`, `<nav>`, `<footer>`, `<header>`, `<aside>`, `<iframe>`, `<form>`, `<button>`, `<svg>`, `<figure>`. Converts block elements to Markdown (headings, paragraphs, blockquotes, lists, tables). Decodes HTML entities (named and numeric). Collapses excessive whitespace.

### Metadata Extraction

Two sources, in priority order:

1. **HTML metadata** (for web sources): `og:title`, `<title>` tags, `name="author"` meta, `article:author`
2. **AI metadata enrichment** (via `Get-AIMetadata`): Uses the AI model to extract title, authors, date, topics, POV tags from the document content. Skippable with `-SkipAiMeta`.

### Output Structure

```
sources/<doc-id>/
  raw/<original-filename>     # Original file (binary or text)
  snapshot.md                 # Markdown conversion with provenance header
  metadata.json               # { summary_status: "pending", title, authors, pov_tags, topic_tags }
```

Document ID is generated by `New-Slug` — a URL-safe slug derived from the title (e.g., `ai-safety-benchmarks-2026`).

## 2. Chunking Mechanism

**Implementation:** `Split-DocumentChunks` (`scripts/AITriad/Private/Split-DocumentChunks.ps1`)

### When Chunking Triggers

The system estimates the document's token count using a linear approximation: **1 token = 4 characters**. Documents exceeding **20,000 estimated tokens** (~80,000 characters) are chunked. Smaller documents are processed in a single AI call.

### Algorithm: Three Phases

**Phase 1 — Semantic Boundary Detection**

The document is split at Markdown heading boundaries using the regex `(?m)^(?=#{2,4}\s)`. This preserves section integrity — chunks never break mid-section. If no headings are found, the document is split on double newlines (paragraph breaks).

**Phase 2 — Greedy Packing**

Sections are accumulated sequentially until adding the next section would exceed `MaxChunkTokens`. When a single section exceeds `MaxChunkTokens`, it is sub-split on paragraph breaks. This is greedy packing, not bin packing — sections are processed in document order.

**Phase 3 — Runt Merging**

If the final chunk falls below `MinChunkTokens`, it is merged into the preceding chunk. This prevents thin trailing chunks that lack sufficient context for meaningful analysis.

### Key Design Decision: No Overlap

Chunks are **non-overlapping**. There is no sliding window. Each piece of text appears in exactly one chunk. The rationale: the merge step deduplicates by claim key, so overlap would just create duplicate work without improving extraction quality.

### Parameters

| Parameter | Default | Purpose |
|-----------|---------|---------|
| `MaxChunkTokens` | 15,000 | Maximum tokens per chunk |
| `MinChunkTokens` | 2,000 | Minimum tokens before merging into previous |
| `ChunkThresholdTokens` | 20,000 | Document size that triggers chunking |
| Token estimation | 1 token = 4 chars | Linear approximation (no tokenizer) |

### Example

A 50,000-word document (~200,000 chars, ~50,000 tokens):
- Exceeds 20,000 token threshold → chunking triggered
- Split at heading boundaries into sections
- Packed into ~3-4 chunks of ~15,000 tokens each
- Any final chunk under 2,000 tokens merged upward

## 3. AI Summarization

### 3a. Single-Call Path (documents <= 20,000 tokens)

**Entry point:** `Invoke-POVSummary` (`scripts/AITriad/Public/Invoke-POVSummary.ps1`)

The entire document is sent to the AI model in a single call, along with the full taxonomy for reference.

**Prompt assembly:**

```
[System prompt: extraction instructions + density targets]
[Taxonomy JSON: all 4 files — accelerationist, safetyist, skeptic, situations]
[Policy registry: up to 5KB from policy_actions.json]
[Output schema: exact JSON structure expected]
[Document header: title, POV tags, topic tags]
[Full document snapshot]
```

**AI parameters:**

| Parameter | Value |
|-----------|-------|
| Model | `gemini-3.1-flash-lite-preview` (default, configurable via `-Model`) |
| Temperature | 0.1 (highly deterministic) |
| MaxTokens | 32,768 |
| JsonMode | Enabled (forces valid JSON response) |
| Timeout | 120 seconds |

**Density-scaled extraction targets:**

The system scales extraction targets based on document length to ensure thorough coverage of longer documents and prevent over-extraction from short ones.

| Target | Formula | 5,000-word doc | 15,000-word doc |
|--------|---------|----------------|-----------------|
| Key points MIN (per POV) | MAX(3, wordCount / 500) | 10 | 30 |
| Key points MAX (per POV) | MAX(8, wordCount / 200) | 25 | 75 |
| Factual claims MIN | MAX(3, wordCount / 800) | 6 | 19 |
| Factual claims MAX | MAX(8, wordCount / 300) | 17 | 50 |
| Unmapped concepts MIN | MAX(2, wordCount / 2000) | 2 | 8 |
| Unmapped concepts MAX | MAX(5, wordCount / 800) | 6 | 19 |

**Density validation and retry:**

After the AI response, the system counts key_points, factual_claims, and unmapped_concepts. If any fall below the minimum thresholds, a single retry is attempted with a "density nudge" appended to the prompt specifying the exact shortfalls (e.g., "Expected at least 10 key_points for accelerationist, got 6").

**JSON repair:** If the response is truncated (common with long outputs), `Repair-TruncatedJson` attempts to close open brackets and quotes to salvage a partial response.

### 3b. Chunked Path (documents > 20,000 tokens)

**Implementation:** `Invoke-ChunkedSummary` in `Invoke-DocumentSummary.ps1`

Each chunk is processed independently with the full taxonomy context. Results are merged afterward.

**Per-chunk AI parameters:**

| Parameter | Value |
|-----------|-------|
| MaxTokens | 65,536 (larger than single-call, since chunks need room) |
| Temperature | 0.1 |
| JsonMode | Enabled |
| Timeout | 300 seconds |
| MaxRetries | 3 (with delays of 5, 15, 45 seconds) |

Chunks are processed **sequentially** (not in parallel) to respect API rate limits. Each chunk receives a header: `--- DOCUMENT SECTION N OF M ---`.

The chunk prompt uses `pov-summary-chunk-system.prompt`, which includes the instruction: "Extract EVERY distinct point in THIS SECTION ONLY. A later merge step will deduplicate across sections."

### Chunk Merge Logic

**Implementation:** `Merge-ChunkSummaries` (`scripts/AITriad/Private/Merge-ChunkSummaries.ps1`)

Results from all chunks are merged with deduplication:

| Element | Deduplication Key | Scope |
|---------|------------------|-------|
| Key points | `taxonomy_node_id | point[0:80]` (case-insensitive) | Per POV camp |
| Factual claims | `claim_label` (lowercase), fallback `claim[0:60]` | Global |
| Unmapped concepts | `suggested_label` (lowercase) | Global |

The algorithm uses HashSets for O(1) lookup. First occurrence wins — later duplicates are discarded without aggregation.

Density validation runs on the merged result but issues **warnings only** (no retry for chunked results, since re-running all chunks is expensive).

## 4. Claim Extraction — What the AI Produces

The AI extracts three categories of output per document:

### Key Points (per POV camp)

Each of the three POV camps (accelerationist, safetyist, skeptic) gets an independent set of key points. The AI reads the document through each POV's lens.

```json
{
  "stance": "aligned",           // strongly_aligned|aligned|neutral|opposed|strongly_opposed|not_applicable
  "taxonomy_node_id": "acc-desires-001",  // exact node ID or null if unmapped
  "category": "Desires",         // Desires|Beliefs|Intentions
  "point": "3-6 sentences analyzing this claim from the POV perspective",
  "verbatim": "1-5 sentences copied word-for-word from the document",
  "excerpt_context": "Page/section where the excerpt appears"
}
```

**Category disambiguation rules given to the AI:**
- **Desires** — Normative commitments: "should" statements, end-states, risk thresholds, principles
- **Beliefs** — Empirical claims: measurements, predictions, historical facts, observations
- **Intentions** — Reasoning strategies: policy approaches, rhetorical methods, logic models, frameworks

**Stance** indicates how the document's position relates to the taxonomy node: does it support, oppose, or qualify that position?

### Factual Claims

Cross-POV empirical assertions extracted from the document. These are the raw material for conflict detection.

```json
{
  "claim": "2-5 sentences describing a specific empirical assertion",
  "claim_label": "5-10 word short title",
  "doc_position": "supports",     // supports|disputes|neutral|qualifies
  "temporal_scope": "predictive", // current_state|predictive|historical|timeless
  "temporal_bound": "by 2030",    // date string or null
  "potential_conflict_id": null,   // existing conflict ID if matches known disagreement
  "linked_taxonomy_nodes": ["acc-beliefs-003", "saf-beliefs-007"]
}
```

### Unmapped Concepts

Claims that don't map to any existing taxonomy node. These are candidates for new nodes.

```json
{
  "suggested_label": "5-8 word plain-language label",
  "suggested_description": "Genus-differentia format: 'A Belief within skeptic discourse that...'",
  "concept": "2-5 sentences grounding this in the document",
  "suggested_pov": "skeptic",
  "suggested_category": "Beliefs",
  "reason": "Why this deserves a new taxonomy node"
}
```

For concepts suggested as situations (cross-POV), the AI also produces three interpretation fields — one per POV.

## 5. Taxonomy Mapping — How Claims Match Nodes

Taxonomy mapping is performed **by the AI model during summary generation**, not as a separate post-processing step. The full taxonomy (all four files, ~200+ nodes) is injected into the prompt, and the AI is instructed to:

1. **Search all four taxonomy files** (accelerationist, safetyist, skeptic, situations) for a matching node
2. **Set `taxonomy_node_id`** to the exact node ID (e.g., `acc-desires-001`, `saf-beliefs-002`, `cc-003`)
3. **Cross-POV search**: Before declaring a concept unmapped, search all POVs — a safetyist claim might map to an accelerationist node that addresses the same phenomenon from a different angle
4. **Set `taxonomy_node_id: null`** only if no existing node captures the concept

There is no embedding similarity, NLI, or vector search in the mapping step. The AI model performs semantic matching against the full taxonomy provided in the prompt context. This works because:
- The taxonomy is small enough (~200 nodes, ~50KB of JSON) to fit in the prompt
- The AI model has the full node descriptions available for semantic comparison
- Node IDs are structured (prefix encodes POV and category), which helps the model navigate

### Policy Mapping

When a document mentions policy actions (regulation, legislation, standards), the AI maps them to entries in the policy registry (`policy_actions.json`) using `pol-NNN` IDs. The registry is capped at 5KB in the prompt to avoid context bloat.

## 6. Unmapped Concept Detection and Proposal

### Detection (During Summary)

Unmapped concepts are detected during the summary generation step. When the AI cannot find a matching taxonomy node for a key point or factual claim, it:

1. Sets `taxonomy_node_id: null` on the key point
2. Adds an entry to `unmapped_concepts[]` with a suggested label, description, POV, and category

### Aggregation (Health Analysis)

`Get-TaxonomyHealthData` aggregates unmapped concepts across all summaries:
- Filters by frequency (concepts appearing in >= 2 documents)
- Ranks by frequency, takes top 30
- Reports alongside other health metrics: orphan nodes, most-cited nodes, high-variance nodes, POV coverage balance

### Proposal Generation

`Invoke-TaxonomyProposal` (`scripts/AITriad/Public/Invoke-TaxonomyProposal.ps1`) takes the aggregated health data and generates structured proposals for taxonomy improvements:

| Parameter | Value |
|-----------|-------|
| Temperature | 0.3 (slightly creative — higher than extraction) |
| MaxTokens | 16,384 |
| JsonMode | Enabled |
| Timeout | 120 seconds |

**Proposal types:**
- `NEW` — Create a new taxonomy node for a frequently unmapped concept
- `SPLIT` — Split an overloaded node into more specific children
- `MERGE` — Combine near-duplicate nodes
- `RELABEL` — Rename a node whose label doesn't match its actual usage

Proposals are saved to `taxonomy/proposals/proposal-{timestamp}.json` for human review. They are never auto-applied.

## 7. Conflict Detection

**Implementation:** `Find-Conflict` (runs automatically after summary generation)

After a summary is written, the system examines its `factual_claims` for cross-POV disagreements:

1. Each claim has `linked_taxonomy_nodes` — the nodes it relates to
2. If two claims from different POV perspectives contradict each other (opposing `doc_position` values on overlapping nodes), a conflict is created or updated
3. Conflicts are stored in `conflicts/<conflict-id>.json` with:
   - The disputed claim label and description
   - Instance array: which documents support, dispute, or qualify each side
   - Linked taxonomy nodes
   - Status: `open`, `resolved`, or `wont-fix`

## 9. What Is NOT Done

For clarity, here is what the pipeline does **not** do:

- **No embedding similarity for taxonomy mapping.** The AI model performs semantic matching using the full taxonomy in the prompt context. There is no vector database or cosine similarity step during ingestion.
- **No NLI (Natural Language Inference)** during ingestion. NLI is used in the Electron app for semantic search, not in the PowerShell pipeline.
- **No automatic taxonomy modification.** Unmapped concepts generate proposals for human review. The taxonomy is never auto-updated by the ingestion pipeline.
- **No cross-document aggregation during ingestion.** Each document is summarized independently. Cross-document analysis (conflict detection, health metrics) runs as a separate step.
- **No sliding window chunking.** Chunks are non-overlapping with semantic boundary detection.
- **No parallel chunk processing.** Chunks are processed sequentially to respect API rate limits.
