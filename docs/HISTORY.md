# AI Triad Research — Project History

A narrative record of significant decisions, shifts, and milestones. Not a changelog — read this to understand *why* the project is the way it is.

## 2026-04-03 — QBAF Diagnostics approved

The QBAF engine computes strength scores — now users need to see and explore them (t/167). Four visualization modes: a strength timeline chart showing how claim strengths evolve across debate turns, per-claim detail cards showing the attack/support breakdown, inline delta badges on the transcript (▲/▼ showing net impact per turn), and a What-If mode for counterfactual analysis ("what would happen to this claim's strength if we removed this attacker?"). The key design constraint: no new AI calls. Everything is deterministic DF-QuAD math on the existing QBAF graph — the engine already computed the data, this just makes it visible and explorable. Six tickets (t/168–t/173), all Taxonomy Editor UI work. Per the Computational Linguist's proposal in e/22.

## 2026-04-03 — Concession Harvesting approved

A new feedback loop between debates and the taxonomy (t/150). When a debater concedes a point — and particularly when the same concession recurs across multiple debates — the system now accumulates that signal and surfaces it for human review as a taxonomy update candidate: qualify a node's description, weaken its steelman, or retire it entirely. This is the first feature that lets debate outcomes flow back into the taxonomy's structure, not just its metadata. Two phases: automatic bookkeeping (track concession frequency and patterns) then harvest dialog integration (present candidates in the existing review workflow). Follows the project's consistent pattern of no auto-apply — humans approve all taxonomy mutations. Per the Computational Linguist's proposal in e/21.

## 2026-04-03 — Debate clarification UX redesigned

The debate setup's clarification phase — where the AI asks the user follow-up questions before debaters begin — is getting a UX overhaul (t/137). Previously the user typed free-text answers, which was friction-heavy and produced unstructured input that debaters couldn't reliably consume. The new design has the AI generate clickable option pills for each clarification question, with a free-text fallback for anything the options don't cover. Three tickets across Shared Lib (prompt changes to generate options) and Taxonomy Editor (pill UI + synthesis wiring). A small but user-facing quality-of-life improvement that also produces more structured data for the debate pipeline to work with.

## 2026-04-03 — QBAF calibration benchmark resolved — hybrid scoring adopted

The QBAF calibration gate (Q-0, t/115) took 25 messages across 7 participants and 4 prompt iterations to resolve — the longest design thread in the project (e/19). The outcome upends the original assumption that AI could score all claim types uniformly. The Computational Linguist designed a BDI-aware decomposed rubric with different sub-criteria per category: Beliefs scored on evidential grounding and specificity, Desires on normative clarity and stakeholder scope, Intentions on mechanism detail and feasibility. AI scoring achieved acceptable correlation for Desires (r=0.65) and Intentions (r=0.71), but failed on Beliefs — LLMs couldn't reliably assess evidential grounding because it requires verifying claims against external sources, not just evaluating internal coherence. The solution is hybrid scoring: AI rubric for Desires and Intentions, human-assigned for Beliefs. This is a meaningful finding beyond the project: evidential grounding assessment is fundamentally harder for LLMs than values or mechanism assessment. QBAF Phase 2 is now unblocked.

## 2026-04-03 — Prompt Quality Baseline — first systematic prompt review

The Computational Linguist reviewed all 39 prompt templates across the project and the results are both reassuring and sobering (t/121). The debate prompts scored A- ("highest quality in the project — linguistically precise, dialectically sophisticated"), while chat prompts scored B- ("weakest area"). Eight high-priority issues were identified, clustering around three themes: ~40% of prompts lack few-shot examples (degrading cross-model consistency), an exhaustiveness directive in the summarization prompt that actively invites hallucination ("if the document warrants more than the target maximum, include them"), and attention dilution from large unfiltered context payloads. The most architecturally significant finding: the synthesis prompt tries to do 9 tasks in one shot (agreement, disagreement, cruxes, questions, taxonomy refs, argument map, preference evaluation, policy implications), risking attention collapse on later tasks. Also notable: the Relevance Engineering ★ tiering markers aren't mentioned in the prompt instructions that consume them — the backend does the work but the model doesn't know to use it. Eight tickets (t/122–t/129) assigned across PowerShell, Shared Lib, and Taxonomy Editor. Full report: `roles/comp-linguist/prompt-review-2026-04-03.md`.

## 2026-04-02 — Advanced Pipeline Techniques approved

Three research-informed pipeline initiatives approved (t/116, e/16). **CHESS** (hierarchical context selection) replaces full-taxonomy injection (200+ nodes) with dynamic branch assembly — pre-classify documents, inject only relevant branches, with a catch-all safety margin. **FIRE** (confidence-gated iterative extraction) replaces single-shot summarization with an agent loop: assess confidence per claim, search for evidence when uncertain, terminate early when queries converge. This subsumes the earlier multi-step decomposition ticket (PP-4/t/87). **TaxoAdapt** (automated taxonomy refinement) proposes structural changes based on mapping density signals — but the design review's most significant outcome was a unanimous block on auto-apply. Every reviewer independently flagged autonomous taxonomy mutation as too risky; v1 is strictly propose-review-apply with batch approval UI, max 3 node changes per proposal, and mandatory dry-run with rollback journal. The sequencing interleaves with QBAF: CHESS after QBAF Phase 1, FIRE after Phase 2, TaxoAdapt last. These represent the project pushing toward self-improving infrastructure — not just analyzing documents, but improving how it analyzes them.

## 2026-04-02 — QBAF shipped — formal argumentation replaces binary conflict detection

The project's most theoretically ambitious initiative is effectively complete: 20 of 21 QBAF tickets delivered (t/95). The DF-QuAD gradual semantics engine computes continuous argument strength scores across the entire claim graph — every claim now carries a mathematically grounded strength that accounts for all its supporters and attackers. All seven use cases are live: conflict detection, live debate (strength scores update after each turn), harvest integration, convergence tracking, taxonomy health, fact-check weighting for web evidence, and POVer Chat. The multi-agent combinator reconciles cross-POV claims with an NLI entailment gate (added after the Risk Assessor flagged cosine-only merging as insufficient). The last ticket — formal deprecation of `Find-Conflict` — was intentionally deferred; the old system runs in parallel as a safety net.

The calibration story (Q-0) deserves its own note — see the entry above. It nearly derailed the timeline: the initial rubric scored r=-0.12 (worse than random), the Computational Linguist identified a BDI category bias in the rubric design, and the team iterated through 4 prompt versions over 26 emails before landing on hybrid scoring. The lesson: you can't build a one-size-fits-all quality rubric for claims that operate at fundamentally different epistemological levels (empirical evidence vs. normative values vs. strategic mechanisms).

## 2026-04-02 — Node ID space migration completed — terminology alignment finished

The third and final terminology migration shipped: ~15,000 node IDs across ~1,800 files, the largest single migration in the project's history (t/78). Node IDs now match their conceptual names — `acc-goals-001` became `acc-desires-001`, `saf-data-002` became `saf-beliefs-002`, `skp-methods-001` became `skp-intentions-001`, `cc-001` became `sit-001`. The Phase 0 centralization strategy proved its worth: 26 scattered prefix-detection sites were consolidated into `lib/nodeIdUtils.ts`, which meant the actual data and code migration was a single-constant change instead of a fragile multi-file edit. TAXONOMY_VERSION bumped to 3.0.0.

This closes the terminology alignment arc that began with the BDI migration (t/6, display labels), continued through the Situations migration (t/13, type names and UI), and ended here with the ID layer (t/78, storage keys). Each migration was larger and more ambitious than the last — 41 files, then 100+ files, then 1,800 files — but each also benefited from lessons learned: the 4-phase audit-then-migrate pattern, pre/post tagging, and now prefix centralization. The codebase is fully aligned from user-facing labels down to storage identifiers.

## 2026-04-02 — Prompt Inspector Phase B shipped — prompt pipeline fully user-controllable

The Prompt Inspector is now feature-complete. Phase B (t/72) delivered the configuration controls that Phase A's read-only view was building toward: cosine similarity thresholds, max node counts, BDI category toggles, vulnerability and fallacy caps, policy limits, temperature, model selection, and response length — all configurable per-prompt with live volume and token count updates as sliders move. Settings persist per debate/chat session with workspace-level defaults as fallback. This closes the loop on the original user request: the ability to see *and* configure how AI prompts are assembled. The journey from "prompts are a black box" (before Phase A) to full transparency and control took three building blocks: Relevance Engineering (made parameters tunable in the backend), Phase A (made the pipeline visible), Phase B (put controls in the user's hands). Phase C (live auto-select during debates, token budget visualization) is deferred as polish.

## 2026-04-02 — Prompt Inspector Phase A shipped

The read-only Prompt Inspector is live. Users can now see exactly what the AI sees — all 27 prompt templates browsable by group (debate setup, debate turns, analysis, moderator, chat, taxonomy, research, PowerShell backend), each showing its data pipeline and raw template with highlighted placeholders. The "Generate Preview" feature assembles a fully rendered prompt using real data from an active debate or chat session, so the user can inspect the exact text before it's sent. This is the transparency layer the project has been missing: until now, prompt assembly was a black box. Phase B (configurable controls for node caps, thresholds, BDI filters) is deferred — it depends on the Relevance Engineering parameters being wired through, which is now complete (t/19), so it's unblocked whenever the team is ready.

## 2026-04-02 — Electron upgrade completed (35 → 41)

All three Electron apps jumped from 35.0.0 to 41.1.1 in a single direct upgrade (not incremental), along with electron-builder 25 → 26. The analysis-first approach paid off: the audit identified only low-medium risk — a clipboard API migration across 7 files and a PDF export fix. All 63 IPC channels verified working. The upgrade eliminated 11 known vulnerabilities, dropping to zero. The team chose a direct jump over incremental version-stepping after the breaking changes audit showed the surface area was manageable. A follow-on toolchain refresh (TypeScript 6, Vite 8, Zod 4) was identified during the compatibility analysis but deferred as a separate epic.

## 2026-04-02 — Ontological compliance epic launched

The project is formalizing its relationship with three academic frameworks it has been informally adopting: DOLCE (genus-differentia descriptions, subsumption), BDI (Belief-Desire-Intention categories), and AIF (Argumentation Interchange Format edge types, node scopes, argument maps). Until now these were conventions documented in AGENTS.md but not enforced — an audit found only 5.8% of node descriptions actually match the genus-differentia pattern the project claims to follow. The new epic (t/36) closes this gap in five phases: schema hardening (Zod schemas that reject non-compliant data), audit tooling (`Test-OntologyCompliance` cmdlet), prompt enforcement (AI prompts that produce compliant output), runtime validation (reject violations on save), and finally data migration to bring existing content into compliance. Phases 1–2 can start immediately; the rest sequence after. This is the project's first serious investment in data quality enforcement as opposed to data generation.

## 2026-04-02 — SBOM cmdlet approved

The project is getting a Software Bill of Materials tool: `Get-AITSBOM` (t/35). This is the first cmdlet that looks *outward* at the project's dependency health rather than inward at its content. It enumerates every dependency across all component types — PowerShell modules, npm packages, Python packages, system tools (pandoc, gs), AI models from the registry, and even JSON schemas. `-CheckUpdates` checks freshness against package registries; `-Update` handles upgrades with confirmation. Output formats include the usual (Table, JSON, CSV) plus industry-standard CycloneDX and SPDX for compliance. Assigned to the PowerShell profile.

## 2026-04-02 — Prompt Inspector approved

A new feature was specced to give the user visibility and control over the AI prompt pipeline. The system currently assembles prompts behind the scenes — injecting taxonomy nodes, vulnerabilities, fallacies, commitments, and other context based on hardcoded thresholds. The user can't see what the AI actually receives, much less tune it. The Prompt Inspector (t/34) fixes this in three phases: first a read-only catalog and preview of all 25+ prompts, then configurable controls (sliders for node caps, similarity thresholds, BDI category toggles), and finally live context-awareness during active debates. It's the user-facing complement to Relevance Engineering — RE provides the scoring engine, the Inspector provides the steering wheel. Blocked by the Situations migration; Phase B additionally blocked by RE. Spec in e/9#2.

## 2026-04-02 — Relevance Engineering completed

All 8 tickets delivered (t/20–t/27). This was the most architecturally significant change to the debate system's AI integration since debates were introduced. The core problem: the formatting layer was flattening scored, structured taxonomy data into undifferentiated text blocks, losing the signal that relevance scoring was meant to provide. The fix touched every stage of context assembly. Taxonomy nodes are now tiered primary/supporting with visual markers. Vulnerabilities are split from fallacies and topic-filtered instead of dumped in bulk. Instructions scale by response length. Policies are filtered to the top 10 by relevance. Established points are prioritized by argument relevance. Document analysis is pre-filtered by embedding similarity. BDI prompt headers use grounded labels instead of generic ones. Eight commits by the Shared Lib profile, all scoped to `lib/debate/`. The Prompt Inspector (t/34) can now expose these as user-configurable controls.

## 2026-04-02 — Cross-Cutting → Situations migration begun

Following the BDI migration's success, the team launched a second large-scale terminology migration: renaming "cross-cutting" to "Situations" throughout the codebase. The motivation is conceptual clarity — "cross-cutting" is developer jargon that describes a structural relationship, while "Situations" better captures what these nodes actually represent: real-world conditions where multiple POVs converge or collide. The scope is larger than BDI: 100+ files, 500+ instances, 7 file renames, a JSON key rename (`cross_cutting_refs` → `situation_refs`), and a public cmdlet rename with deprecation wrapper. The `cc-` ID prefix was preserved at this stage to avoid a data migration. (Later migrated to `sit-` in the Node ID Space migration, t/78.) Same 4-phase audit-then-migrate approach that proved itself with BDI. Plan in e/6.

## 2026-04-02 — BDI terminology migration completed

The project's first large-scale terminology migration shipped successfully. The taxonomy's cognitive model categories — previously Data/Facts, Goals/Values, Methods/Arguments — were renamed to Beliefs, Desires, and Intentions to align with the established BDI (Belief-Desire-Intention) framework from agent theory. The migration touched 41+ files and 270+ instances across both repos, executed in four phases: audit, shim layer, bulk rename, shim removal. Node ID slugs were deliberately preserved at this stage to avoid a data migration — a pragmatic decision that decoupled display labels from storage keys. (The ID slugs were later migrated in the Node ID Space migration, t/78.) Both repos were tagged pre- and post-migration, and `TAXONOMY_VERSION` was bumped to 2.2.0. The full plan thread (e/1) and approval (e/2) document every design decision.

## 2026-04-02 — Orca team stood up; error handling standardized

The project adopted Orca as its AI team management system. Nine profiles were created: four code-owning profiles (Taxonomy Editor, PowerShell, Shared Lib, Orca Support) and five role-based profiles (Technical Lead, SRE, Product Manager, Risk Assessor, Diagnostics, Historian). The team can now communicate via pings and email, with agents waking each other for coordination.

Alongside this, a mandatory error handling standard was established. Two shared utilities — `New-ActionableError` (PowerShell) and `ActionableError` (TypeScript) — enforce structured diagnostics: every unrecoverable error must state its goal, problem, location, and specific next steps. An `error-handling-auditor` subagent was created to enforce compliance incrementally as code is touched.
